## 🌐 About OSLKA

**Open-Source Linux Knowledge Academy (OSLKA)** is a free, open, and globally accessible knowledge platform built by the community — for the community.

We believe that **Linux education should have no barriers**. Whether you're typing your very first `ls` command or diving deep into kernel memory management, OSLKA offers a structured, peer-reviewed, and continuously evolving knowledge base that grows with you.

### 💡 Why OSLKA Exists

The Linux ecosystem is vast, powerful, and constantly evolving — yet structured, accessible, and community-driven learning resources remain fragmented. OSLKA was created to:

- 🔓 Break down barriers to Linux and open-source education
- 🗂️ Consolidate scattered knowledge into a single, structured repository
- 🤝 Foster a culture of contribution, peer learning, and mentorship
- 🎓 Support students, self-learners, and researchers with production-grade documentation
- 🔬 Build a bridge between academic Linux research and real-world engineering practice

### 🌍 Global Vision & Objectives

| Objective | Description |
|-----------|-------------|
| 🌱 **Structured Learning** | From Linux basics to Kernel Development — a clear, progressive path |
| 📚 **Free Knowledge Base** | All content is free, open-source, and community-reviewed |
| 🔬 **Research Culture** | Promote academic and applied Linux research |
| 🌐 **Multi-Language Docs** | Documentation in multiple languages for global access |
| 🤝 **Mentorship Network** | Connect beginners with experienced Linux professionals |
| 🏆 **Recognition System** | Reward contributors with meaningful, skill-based ranks |

## 🗺️ Learning Roadmap

> A structured, progressive learning path — from zero to kernel developer.

<details open>
<summary><strong>🟢 Foundation Track</strong></summary>

```
🔰 Stage 1 — Linux Introduction
   ├── What is Linux & Open Source?
   ├── Linux Distributions Overview
   ├── Linux History & Philosophy
   └── Installing Linux (VMs, WSL, Bare Metal)

📁 Stage 2 — Linux Fundamentals
   ├── File System Hierarchy (FHS)
   ├── Essential Commands (ls, cp, mv, rm, find, grep)
   ├── Permissions & Ownership (chmod, chown, ACLs)
   └── Package Management (dnf, apt, rpm)

🗂️ Stage 3 — File Systems
   ├── ext4, XFS, Btrfs, tmpfs
   ├── Mounting & Automounting
   ├── LVM (Logical Volume Manager)
   └── RAID Configuration
```

</details>

<details>
<summary><strong>🔵 Intermediate Track</strong></summary>

```
👤 Stage 4 — User & Group Management
   ├── Users, Groups, UIDs, GIDs
   ├── sudo, su, PAM
   └── LDAP & Directory Integration

💻 Stage 5 — Shell Scripting
   ├── Bash Scripting Fundamentals
   ├── Advanced Bash (arrays, functions, traps)
   ├── Python for Linux Admins
   └── Automation with Ansible

🌐 Stage 6 — Networking
   ├── TCP/IP, DNS, DHCP
   ├── NetworkManager, nmcli, ip
   ├── Firewalld & nftables
   └── Network Troubleshooting

💾 Stage 7 — Storage Management
   ├── Disk Partitioning (fdisk, parted)
   ├── LVM Advanced Operations
   ├── NFS & Samba
   └── iSCSI & Multipathing

🔒 Stage 8 — Linux Security
   ├── Hardening Checklist
   ├── SSH & PKI
   ├── Audit & Logging
   └── Vulnerability Management

🛡️ Stage 9 — SELinux
   ├── SELinux Architecture
   ├── Policies & Contexts
   ├── Troubleshooting (audit2allow, sealert)
   └── Writing Custom Policies
```

</details>

<details>
<summary><strong>🔴 Advanced Track</strong></summary>

```
🐳 Stage 10 — Containers (Docker / Podman)
   ├── Container Fundamentals
   ├── Dockerfile & Containerfile
   ├── Rootless Containers
   └── Container Networking & Storage

☸️ Stage 11 — Kubernetes
   ├── Kubernetes Architecture
   ├── Pods, Deployments, Services
   ├── Storage & Networking
   └── Helm & Operators

🔴 Stage 12 — OpenShift
   ├── OpenShift vs Kubernetes
   ├── Routes, DeploymentConfigs

   └── OpenShift Security (SCC)

   ├── Linux on AWS / Azure / GCP
   ├── Infrastructure as Code
   ├── Cloud-Native Patterns
   └── FinOps for Linux Workloads

   ├── Boot Process (BIOS → GRUB → systemd)
   ├── Process Management & Scheduling
   ├── Memory Management
   └── I/O Subsystem

   ├── syscall() fundamentals
   ├── strace & ltrace
   ├── Kernel-Userspace Interface
   └── eBPF for System Observability

🏗️ Stage 16 — Kernel Architecture
   ├── Kernel Source Tree
   ├── Subsystems Overview
   ├── Kernel Data Structures
   └── Kernel Debugging (GDB, KGDB, QEMU)

🧪 Stage 17 — Kernel Development
   ├── Setting Up the Kernel Dev Environment
   ├── Writing Kernel Modules
   ├── Submitting Patches to LKML
   └── Kernel Testing (KUnit, kselftest)
```

</details>

## 🔬 Research Hub

OSLKA is not just a learning platform — it is a **research community**. We encourage academic and applied research contributions across the following domains:

<details>
<summary><strong>🔐 Linux Security Research</strong></summary>

- Exploit mitigation techniques (KASLR, SMEP, SMAP, CFI)
- Linux kernel CVE analysis and patch reviews
- Red team / Blue team Linux techniques
- Kernel hardening and audit frameworks
- PAM, NSS, and authentication research

</details>

<details>
<summary><strong>⚙️ Kernel Research</strong></summary>

- Memory management subsystem (buddy allocator, slab/slub, NUMA)
- CPU scheduler evolution (CFS, EEVDF)
- VFS (Virtual File System) internals
- Device driver architecture
- Real-time Linux (PREEMPT_RT)

</details>

<details>
<summary><strong>🧬 eBPF Research</strong></summary>

- eBPF verifier internals
- XDP (eXpress Data Path) networking
- BPF CO-RE (Compile Once, Run Everywhere)
- Observability tools (bpftrace, Cilium, Falco)
- eBPF for security policy enforcement

</details>

<details>
<summary><strong>🖥️ Virtualization & KVM / KubeVirt</strong></summary>

- KVM hypervisor architecture
- QEMU internals and virtio
- KubeVirt: running VMs in Kubernetes
- Live migration research
- Hardware-assisted virtualization (Intel VT-x, AMD-V)

</details>

<details>
<summary><strong>☁️ Cloud-Native Technologies</strong></summary>

- OCI container runtime specifications
- Container networking (CNI plugins, eBPF-based networking)
- Kubernetes scheduler extenders and custom controllers
- Operator pattern research
- Service mesh (Istio, Linkerd) internals

</details>

<details>
<summary><strong>🤖 AI on Linux</strong></summary>

- GPU passthrough for AI workloads on Linux
- MLOps pipelines on Linux clusters
- AI-assisted kernel debugging
- Running LLMs at scale on Linux infrastructure
- Edge AI on embedded Linux devices

</details>

<details>
<summary><strong>📄 Open-Source Research Papers</strong></summary>

Community-contributed academic papers, white papers, and technical reports are archived in [`/research/papers/`](./research/papers/). All submissions must include proper citations, methodology, and reproducible results.

</details>

## 🤝 Community Contribution

We ❤️ contributions from everyone. OSLKA thrives because of its global community.

### 🚀 How to Contribute

```bash
# 1. Fork the repository
# Click the "Fork" button on GitHub

# 2. Clone your fork
git clone https://github.com/YOUR-USERNAME/linux-knowledge-academy.git
cd linux-knowledge-academy

# 3. Create a feature branch
git checkout -b feature/your-contribution-name

# 4. Make your changes and commit
git add .
git commit -m "docs: add advanced SELinux policy examples"

# 5. Push to your fork
git push origin feature/your-contribution-name

# 6. Open a Pull Request on GitHub
```

### 📋 Contribution Types

| Type | Description | Label |
|------|-------------|-------|
| 📝 **Documentation** | Add or improve learning content | `docs` |
| 🧪 **Lab Exercises** | Create new hands-on labs | `lab` |
| 🔬 **Research** | Submit research findings or papers | `research` |
| 🐛 **Bug Fix** | Fix incorrect information or broken content | `fix` |
| 🌐 **Translation** | Translate content to other languages | `i18n` |
| 💡 **New Topic** | Propose and add an entirely new learning module | `enhancement` |

### 📌 Pull Request Guidelines

- Follow the [Conventional Commits](https://www.conventionalcommits.org/) format
- Reference related issues with `Closes #issue-number`
- Ensure all Markdown renders correctly before submitting
- Add yourself to the `CONTRIBUTORS.md` file
- All PRs require at least **one review** from a maintainer or senior contributor

> 📖 See [CONTRIBUTING.md](CONTRIBUTING.md) for the full contribution guide.
